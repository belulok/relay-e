## Today: a single dev key

For v0.0.1, all `/v1/*` endpoints are protected by a single API key set via `DEV_API_KEY`:

```bash title=".env"
DEV_API_KEY=sk_dev_local_change_me
```

```bash
curl http://localhost:3001/v1/skills \
  -H "Authorization: Bearer sk_dev_local_change_me"
```

Missing or incorrect → `401 unauthorized`.

This is **dev-only**. In production, treat this like a master key and put the engine behind your own auth proxy or API gateway.

## In production — what to do today

Run Relay-E in your private network and front it with one of:

| Approach | When |
|---|---|
| **API Gateway** (Kong, Tyk, AWS API Gateway) | You already have one — issue per-customer keys, gateway forwards to Relay-E with the master key |
| **App-level proxy** | Your existing app's backend proxies user requests to Relay-E, forwarding the user's identity as headers |
| **Cloudflare Workers + Bearer auth** | Edge auth with low overhead |
| **Tailscale / VPC-only** | Internal use only, no public surface |

Example app-level proxy:

```ts title="your-app/api/relay-e/[...path].ts"
export async function POST(req: Request, { params }: { params: { path: string[] } }) {
  // Use your existing user auth
  const user = await authenticateRequestSomehow(req);
  if (!user) return new Response("unauthorized", { status: 401 });

  // Proxy to Relay-E with the master key + user context
  const url = `${process.env.RELAY_E_URL}/v1/${params.path.join("/")}`;
  return fetch(url, {
    method: "POST",
    headers: {
      "authorization": `Bearer ${process.env.RELAY_E_KEY}`,
      "content-type": "application/json",
      "x-tenant-id": user.tenantId,        // forwarded by Relay-E into ctx
      "x-user-id": user.id,
    },
    body: req.body,
  });
}
```

Your customers never see the master key. Per-user attribution flows through `x-tenant-id` and `x-user-id` headers.

## Roadmap

| Coming | Why |
|---|---|
| **Per-tenant API keys** | First-class multi-tenancy without a proxy |
| **Scoped tokens** (skills, sessions, expiry) | Customer-issuable, narrowly-scoped |
| **OAuth (Authorization Code + PKCE)** | Proper SaaS auth |
| **mTLS** | High-compliance customers |
| **JWT support** | Easy integration with Clerk / WorkOS / Auth0 / Cognito |

When per-tenant keys land, the auth header changes shape but the migration is one config flag.

## Common errors

```json
// Missing header
{
  "error": {
    "type": "authentication_error",
    "code": "unauthorized",
    "message": "Missing or invalid API key",
    "request_id": "req_..."
  }
}
```

```json
// Wrong key
{
  "error": {
    "type": "authentication_error",
    "code": "unauthorized",
    "message": "Missing or invalid API key",
    "request_id": "req_..."
  }
}
```

The same code is returned in both cases — don't leak whether the key exists. The `request_id` is your handle for debugging.
